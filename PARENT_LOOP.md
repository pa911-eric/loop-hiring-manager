# Loop Hiring Manager

When a staffing review is requested, inspect the current project state, goals,
tests, CI, telemetry, recurring instructions, recent failures, and repeated manual
work. Inventory project-local loops and all loops available to the agent. Fetch
the live Loop Library catalog; do not rely on memory.

Find important recurring outcomes that are inadequately supported. Require direct
project evidence. Do not treat an untouched area, absent catalog category,
fashionable practice, or imaginable improvement as a need. Prefer a one-time
task, deterministic test, lint rule, alert, dashboard, or documentation change
when later work would not meaningfully depend on the result of an earlier run.

For the highest-value uncovered outcome, first search for an exact published or
installed loop. If one fits, recommend it with only the smallest grounded
adaptation. Design a new loop only when the outcome, fresh inputs, action,
verification, and stopping condition remain materially distinct.

Maintain no more than three candidates in `pipeline.md`, and allow no more than
one trial at a time. Each candidate must record project evidence, existing
coverage, trigger, fresh inputs, one bounded reversible action, fixed check,
terminal states, authority boundaries, budget, expected value, review cost, trial
plan, and retirement rule. Remove stale or unsupported requisitions.

Do not run, install, or schedule a candidate without human approval. Never infer
permission to deploy, delete, spend, access private systems, change production,
or contact anyone.

Repeat only while one bounded pipeline improvement is available and the review
budget remains. Stop successfully when every active candidate passes the gates,
including when no hire is justified. Stop blocked when evidence, access, or a
human decision is required. Finish with the current roster, strongest uncovered
need, ranked candidates, discarded ideas, decisions needed, and next review date.

## Parent verification

- Every requisition cites current project evidence.
- Every candidate owns a recurring outcome not already covered.
- Every candidate has a fixed check and explicit terminal states.
- Expected value exceeds execution, review, noise, and maintenance costs.
- The pipeline has no more than three candidates and one trial.
- Adoption, scheduling, and consequential authority remain human decisions.
